## 2. 레이어별 설명

### Client Layer
| 구성 요소 | 설명 | 비고 |
|-----------|------|------|
| Chat UI | 나비와의 대화 인터페이스 | 핵심 기능 |
| Garden | 과거 대화 + 행동 기록 타임라인 | 로그인 사용자 전용 |
| My Page | 프로필, 설정, 탈퇴 | 계정 관리 |
| Local Storage | 게스트 모드 대화 임시 저장 | 로그인 전까지 사용 |

### Backend Layer (Supabase)
| 구성 요소 | 설명 | 비고 |
|-----------|------|------|
| Auth | Google OAuth 로그인 | 간편 인증 |
| PostgreSQL | Users, Chats, Emotion_Logs, Action_Logs | 핵심 데이터 저장 |
| Storage | 이미지/파일 저장 (필요 시) | 추후 확장용 |

### AI Layer
| 구성 요소 | 설명 | 비고 |
|-----------|------|------|
| Structured System Prompt | 나비 페르소나, 말투 규칙, 금지사항 정의 | 고정 |
| 압축 상태 주입 | `bond \| mood \| energy \| recent` | 토큰 효율화 |
| LLM | GPT-4o 또는 Claude 3.5 | 응답 생성 |
| Stealth Assessment | 대화 중 자동으로 감정/상태 태깅 | 백그라운드 수집 |

---

## 3. 주요 데이터 흐름

1. **게스트 모드**
   - Local Storage에 대화 임시 저장
   - Stealth Assessment로 감정/상태 태깅 (Emotion_Logs에 저장은 로그인 후)

2. **로그인 순간**
   - Local Storage 데이터 → Supabase로 자동 마이그레이션

3. **대화 턴**
   - 사용자 메시지 → AI Layer (압축 상태 + 시스템 프롬프트와 함께 전달)
   - LLM 응답 생성 + Stealth Assessment 동시 수행
   - 결과 저장 (Chats + Emotion_Logs)

4. **행동 완료**
   - Quick Reply로 행동 완료 → Action_Logs 기록
   - 보상 메시지 + 시각적 피드백

## 4. MVP 아키텍처 핵심 결정 사항

| 항목 | 결정 내용 | 이유 |
|------|-----------|------|
| **상태 전달 방식** | `bond:42 \| mood:피곤 \| energy:보통 \| recent:수면부족` | 토큰 비용 절감 + 제어 가능성 확보 |
| **데이터 수집** | Stealth Assessment | 사용자 부담 최소화, 자연스러운 수집 |
| **하드코딩 여부** | 최소화 (구조화 프롬프트 중심) | 유지보수성과 확장성 확보 |
| **Guest 지원** | Local Storage 사용 | 온보딩 장벽 최소화 |
| **인증** | Google OAuth 단일 지원 | 개발 속도 및 사용자 편의성 |

---

## 5. 향후 확장 방향 (Phase 1+)

- Prompt Caching 도입 (토큰 비용 절감)
- 대화 요약(RAG) + Memory Layer 분리
- Tool / Function Calling을 활용한 상태 조회 최적화
- 더 정교한 개인화 (User Profile 기반 응답)
